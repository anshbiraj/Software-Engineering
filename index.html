Online Food Delivery System · SRS Analysis
"The system should provide fast delivery, accurate orders, and good customer experience."
Ambiguity identification · FR/NFR classification · Refined requirements
(a) Five Ambiguities / Issues Identified
#Ambiguous TermIssue
1"Fast delivery"No time unit specified. "Fast" is subjective — 30 min? 60 min? Varies by user expectations and city geography.
2"Accurate orders"Accuracy is unmeasured. Does it mean item correctness, quantity, packaging, restaurant compliance? No error-rate threshold given.
3"Good customer experience"Entirely subjective. No metric (CSAT score, NPS, response time, UI load time) is defined. Means different things to each stakeholder.
4"Should provide"Weak modal verb. Does not indicate mandatory obligation vs. optional feature. "Shall" implies a binding requirement; "should" is ambiguous.
5No boundary or scopeThe statement mixes delivery logistics, order management, and UX in one sentence — making it impossible to assign responsibility or test individually.
(b) Classification: Functional vs Non-Functional
PhraseClassificationJustification
"Accurate orders" Functional Describes what the system must do — correctly transmit order details from customer to restaurant to delivery agent.
"Fast delivery" Non-Functional A performance/time constraint on the delivery process — a quality attribute, not a system action.
"Good customer experience" Non-Functional A usability and satisfaction quality attribute — relates to how the system behaves, not what it does.
(c) Refined — Measurable & Testable Requirements
FR-01 · Order Accuracy The system shall capture and transmit order details (item name, quantity, special instructions, delivery address) to the restaurant with 99.5% accuracy, validated against the customer's confirmed cart snapshot.
FR-02 · Order Tracking The system shall provide real-time order status updates (Confirmed → Preparing → Out for Delivery → Delivered) with location tracking refreshed every 30 seconds.
NFR-01 · Delivery Time Performance 95% of orders placed within a 5 km radius shall be delivered within 30 minutes of order confirmation under normal traffic conditions.
NFR-02 · Usability A first-time user shall be able to place an order within 3 minutes without external assistance, verified by usability testing with a sample of 20 users achieving ≥85% task completion rate.
NFR-03 · Customer Satisfaction The system shall maintain an average customer satisfaction score (CSAT) of ≥4.2 out of 5.0, measured via post-delivery in-app surveys with a response rate target of ≥30%.

Cloud-Based LMS · Vague Scalability & Quality Requirements
"The system should support many users, be responsive, and ensure high quality."
Critical evaluation · Impact on planning/testing · Quality attribute categorisation · Revised requirements
1. Critical Evaluation — Missing Specification Details
Vague TermMissing Specification Detail
"Many users"No numeric range. 1,000? 100,000? 1 million? No distinction between concurrent vs registered users. No geographic distribution or peak-load scenario defined.
"Be responsive"No latency target. Responsive to what operation? Page load? Video streaming? Quiz submission? Under what network conditions?
"High quality"No quality model referenced (ISO 25010, FURPS+). Does it mean content quality, data accuracy, uptime, or UI polish? Not measurable.
"Should support"Obligation level undefined. "Should" (RFC 2119) implies recommended, not mandatory — problematic for a binding SRS.
2. Impact of Vague Requirements on Planning and Testing
  • Performance planning: Without a user concurrency target, architects cannot make infrastructure decisions — cluster size, database sharding strategy, CDN tier, and caching policies all depend on load projections.
  • Capacity testing: Load testing tools (JMeter, Locust) require a specific target user count and acceptance threshold. "Many users" cannot be parameterised into a test script.
  • Acceptance criteria: QA teams cannot define pass/fail criteria for "responsive" or "high quality" — testing becomes subjective and inconclusive.
  • Cost estimation: Cloud infrastructure cost modelling is impossible without defined concurrent user limits and throughput requirements.
  • SLA definition: Contracts with clients cannot specify penalties or guarantees without numeric thresholds.

E-Commerce System · Security vs UX Trade-off
"The system should be secure and provide a seamless user experience."
Trade-off analysis · NFR classification · Architectural impact · Balanced refined requirements
1. Trade-offs and Conflicts Identified
Security RequirementUX Implication (Conflict)
Multi-Factor Authentication (MFA) on every loginAdds 15–30 seconds to login flow; frustrates returning users; high abandonment rate
Session timeout after 15 minutes of inactivityInterrupts long browsing sessions; poor experience for comparison shoppers
CAPTCHA on checkoutSlows down transaction completion; friction for mobile users with small screens
Encrypted payload processing (server-side)Adds latency to every API call; affects perceived responsiveness
Strict input sanitisation and form validationMay block valid edge-case inputs; error messages can confuse users
Core Conflict: Security increases friction (slows the user, adds steps); seamlessness reduces friction (fewer steps, faster flow). These are architecturally opposing forces that require deliberate trade-off decisions, not a single statement.
2. NFR Classification
AspectNFR Category (ISO 25010)
Secure (authentication, authorisation)Security — Confidentiality, Integrity, Authenticity
Secure (data protection)Security — Non-repudiation, Accountability
Seamless UX (speed)Performance Efficiency — Time Behaviour
Seamless UX (ease)Usability — Operability, User Error Protection
Seamless UX (satisfaction)Usability — User Interface Aesthetics
3. Influence on Architectural and Design Decisions
  • Authentication architecture: Conflict drives adoption of Risk-Based Authentication (RBA) — MFA triggered only for anomalous login patterns (new device, foreign IP), not every login. This satisfies both parties.
  • Session management: Sliding session windows (token refresh on activity) instead of hard timeouts balance security with continuity.
  • API gateway design: Security enforcement (rate limiting, WAF, token validation) moves to the gateway layer so application APIs remain lightweight and fast.
  • Checkout flow: Passive bot detection (device fingerprinting, behavioural analytics) replaces visible CAPTCHA — security without UX friction.
  • Performance budgets: Architectural decision to set a latency budget (e.g., ≤100ms added by security middleware) to prevent security controls from degrading UX below threshold.
4. Balanced Refined Requirement Set
NFR-SEC-01 · Authentication The system shall enforce MFA only for high-risk login events (new device, unrecognised IP, 3+ failed attempts). Standard logins from trusted devices shall complete within 3 seconds total authentication time.
NFR-SEC-02 · Data Encryption All payment and PII data shall be encrypted in transit (TLS 1.3) and at rest (AES-256). Encryption overhead shall not add more than 50ms to checkout API response time.
NFR-UX-01 · Checkout Performance The checkout process (cart → payment confirmation) shall be completable within 4 user interactions and ≤90 seconds for 90% of users on a 4G mobile connection.
NFR-UX-02 · Session Continuity User sessions shall persist for 30 minutes of inactivity and be silently refreshed on activity, with a security re-challenge only after 8 hours of total session duration.

Online Food Delivery Platform · Implementation Hindrance
"The system should provide fast delivery, accurate orders, and a good customer experience."
Problematic terms · Validation/verification difficulty · Requirement type · Reformulation
1. Problematic Terms and Phrases
Term/PhraseWhy It Hinders Implementation
"Fast"No unit of time. Developer cannot choose between optimising for 15 min, 30 min, or 60 min delivery. Leads to arbitrary design choices.
"Accurate orders"Accuracy threshold missing. Is 98% accuracy acceptable? 99.9%? Without a target, no acceptance criterion can be coded or tested.
"Good customer experience"Compound quality attribute with no breakdown. Cannot be translated into a UI design spec, API spec, or SLA clause.
"Should"Ambiguous obligation. In IEEE 830 terminology, "shall" is mandatory; "should" is recommended. This ambiguity causes scope disagreements during delivery.
Single compound sentenceThree distinct requirements (logistics, data accuracy, UX) are combined — non-atomic, cannot be independently assigned, tracked, or tested.
2. Difficulties in Validation and Verification
  • No acceptance test possible: Testers cannot write a test case for "fast" without a threshold. The system cannot pass or fail an undefined criterion.
  • Verification gap: SRS verification checks whether the requirement is correctly specified. These terms cannot pass verification as they violate completeness and precision.
  • Validation gap: Validation checks that the right system is being built. "Good customer experience" does not map to any user story, prototype, or review criteria.
  • Non-traceability: These requirements cannot be traced to design elements or code modules — breaking bidirectional traceability.
3. Requirement Type Classification and Justification
Implied RequirementTypeJustification
Deliver orders (logistics function)FunctionalDescribes an action the system must perform — a behavioural function.
Fast delivery (time constraint)Non-FunctionalA performance quality attribute on the delivery function — not a new action.
Accurate order processingFunctionalDescribes correctness of data processing — what the system does with order data.
Good customer experienceNon-FunctionalA usability and satisfaction quality — how the system behaves, not what it does.
4. Reformulated Requirements with Measurable Criteria
FR-01 · Order Processing The system shall capture, validate, and transmit the complete order (items, quantities, customisations, delivery address) to the assigned restaurant within 10 seconds of customer confirmation.
NFR-PERF-01 · Delivery Time SLA Median delivery time shall not exceed 30 minutes from order confirmation to doorstep delivery for orders within a 5 km service radius, measured across ≥95% of completed orders per month.
NFR-ACC-01 · Order Accuracy Rate The system shall achieve an order accuracy rate of ≥99.2% (ratio of orders delivered as specified to total orders), measured monthly via delivery confirmation and customer dispute logs.
NFR-UX-01 · App Performance The home screen shall render fully within 1.5 seconds on a 4G connection. The order-placement flow shall require no more than 5 steps from item selection to payment confirmation.

  • SLA negotiation: An institution purchasing the LMS cannot hold the vendor accountable without numeric SLAs — "high quality" is not contractually enforceable.
  • Regression baseline: Performance regression tests need a baseline (e.g., page load <2s). Without it, any degradation can be dismissed as "still within acceptable quality."

3. Influence on Architectural and Design Decisions
  • Identity architecture: Adopt Risk-Based Authentication (RBA): low-risk logins (same device, same IP) use password only; high-risk logins (new device, geo anomaly) require OTP. This satisfies security without universal friction.
  • API gateway layer: Centralise security controls (WAF, DDoS protection, rate limiting, JWT validation) at the gateway — keeps application layer fast and decoupled from security policy.
  • Token strategy: Use short-lived access tokens (15 min) with silent background refresh using long-lived refresh tokens (30 days). Users never see re-authentication unless truly at risk.
  • Separation of concerns: Security architects define the threat model and controls; UX team defines interaction flows within those controls. Architecture enforces boundaries so neither team overrides the other.
  • Privacy by design: Minimise PII collected; tokenise payment data via PCI-DSS compliant vault — reduces security surface area without impacting user flow.

NFR-SEC-02 · Payment Data Protection All payment card data shall be tokenised via a PCI-DSS Level 1 compliant vault. Raw card numbers shall never be stored or transmitted through application servers.
NFR-UX-01 · Checkout Latency The end-to-end checkout flow (add to cart → payment confirmation) shall be completable in ≤60 seconds for 95% of users, with ≤5 user interactions required, on a standard 4G mobile connection.

Fintech Digital Payment Platform · Architecture Design
Single-tier monolithic architecture causing performance bottlenecks and deployment delays. Requirements: web+mobile users, 3rd-party gateways, real-time transactions, high availability at peak.
Monolith limitations · Component decomposition · Data flow analysis · Revised architecture proposal
1. Structural Limitations of the Monolithic Architecture
LimitationImpact on System
Single deployment unitAny change (bug fix, feature) requires full system redeployment → deployment delays, higher risk per release, zero-downtime deployment is impractical.
No independent scalingCannot scale only the transaction processing component during peak load — the entire monolith must scale horizontally, wasting resources on idle modules (e.g., reporting).
Single point of failureA crash in one module (e.g., notification service) brings down all payment processing functions — violates high availability requirement.
Tight couplingIntegrating new third-party payment gateways requires changes that ripple through the entire codebase — high change impact, high regression risk.
Technology lock-inAll components share the same runtime, language, and framework. Cannot adopt best-fit technologies for specific concerns (e.g., a high-throughput message broker).
Poor maintainabilityAs codebase grows, interdependencies increase exponentially (Conway's Law effects). Onboarding new developers becomes slow; debugging is complex.
2. Logical Architectural Components and Responsibilities
Client Layer
Web + Mobile Apps

React/Angular web app; iOS/Android native apps. Communicates with backend exclusively via API Gateway. No business logic.

Entry Point
API Gateway

Routes requests, enforces authentication (JWT), rate limiting, SSL termination, load balancing. Single ingress point.

Core Service
User & Auth Service

Registration, login, OAuth2/MFA, session management, role-based access control.

Core Service
Transaction Service

Real-time payment initiation, validation, idempotency checks, transaction state machine management.

Core Service
Payment Gateway Adapter

Adapter pattern for 3rd-party integrations (Stripe, Razorpay, PayPal). Abstracts gateway-specific APIs; supports adding new gateways without code changes in Transaction Service.

Supporting Service
Notification Service

Sends SMS, email, and push notifications asynchronously via message queue. Failures do not block transactions.

Supporting Service
Audit & Reporting

Consumes event stream; builds compliance audit logs, OLAP-style reports, regulatory reports (RBI, SEBI). Read-only replica database.

Infrastructure
Message Broker

Kafka / RabbitMQ for async event-driven communication between services. Decouples producers from consumers.

3. Data Flow and Component Interaction Analysis
  • Synchronous (real-time): Client → API Gateway → Transaction Service → Payment Gateway Adapter → External Gateway → Response propagated back synchronously. Critical path; must be optimised for latency (<500ms end-to-end).
  • Asynchronous (event-driven): Transaction Service publishes a TransactionCompleted event to the message broker. Notification Service and Audit Service consume this event independently — failures in either do not affect transaction processing.
  • Data isolation: Each microservice owns its own database (Database-per-Service pattern). Transaction Service uses a high-performance relational DB (PostgreSQL with read replicas). Audit Service uses a columnar analytics store (Redshift/BigQuery).
  • Bottleneck analysis: The Payment Gateway Adapter is a natural bottleneck — external API rate limits and latency. Mitigation: connection pooling, circuit breakers (Hystrix/Resilience4j), and retry with exponential backoff.
  • Consistency model: Eventual consistency between Transaction Service and Audit Service (acceptable). Strong consistency within Transaction Service using database transactions and idempotency keys (mandatory for payment correctness).
4. Proposed Architecture and Justification
Proposed Architecture: Microservices with Event-Driven Communication, deployed on Kubernetes
Architecture DecisionImprovement Over Monolith
Microservices decomposition Modularity: Each service is independently deployable. A change to the Notification Service does not require redeploying Transaction Service — deployment frequency increases, release risk decreases.
Kubernetes with Horizontal Pod Autoscaling (HPA) Scalability: Transaction Service pods auto-scale from 3 → 30 replicas during peak load within 60 seconds. Other services remain at baseline — resource efficiency improved.
Circuit Breaker pattern (Resilience4j) Fault Isolation: If an external payment gateway is slow, the circuit breaker opens — upstream Transaction Service fails fast and returns an error instead of hanging. Prevents cascading failure.
Kafka event streaming Decoupling: Notification and Audit services consume events at their own pace. A spike in transaction volume buffers in Kafka rather than overwhelming downstream services.
API Gateway (Kong/AWS API GW) Security + Routing: Centralises auth, rate limiting, and routing — simplifies each microservice (no duplicate auth code) and provides a single observability point.
Multi-AZ deployment High Availability: Services deployed across 3 Availability Zones. A zone failure reduces capacity by one-third but does not cause a total outage — supports 99.99% SLA.
Summary Justification The proposed microservices architecture addresses all identified monolith limitations: independent deployability solves deployment delays; targeted scaling solves performance bottlenecks; circuit breakers and multi-AZ deployment solve the single-point-of-failure problem; the adapter pattern solves third-party gateway integration coupling. Each service aligns with a bounded domain context (DDD principle), ensuring long-term maintainability.
