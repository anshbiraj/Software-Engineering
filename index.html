Software and Software Engineering

Pressman · Nature of software · SE layers · Generic process framework · Umbrella activities
1.1 Nature of Software
Definition
Software is: (1) instructions (programs) that when executed provide desired features, function, and performance; (2) data structures that enable programs to adequately store and manipulate information; and (3) documentation that describes the operation and use of programs.

Software is a logical rather than physical system element. It does not wear out — it deteriorates due to change (the bathtub curve does not apply; the failure curve shows increasing failures as changes are made).

Characteristics that distinguish software from hardware
CharacteristicHardwareSoftware
ManufacturingPhysical fabricationEngineering (design/development)
Failure curveBathtub curve — wears outDeteriorates via change, not wear
Custom-builtAssembled from componentsCustom-built (components are emerging)
DeliveryTangible productIntangible — logic & information
Software Change Problem

As software ages and is modified, its structure degrades — producing an "upward sloping" failure rate. This is why software maintenance is the most costly phase (often 60–80% of total lifecycle cost).

Software Failure Curve — Deterioration via Change

1.3 Software Engineering Definition and Layers
IEEE Definition
Software Engineering is the application of a systematic, disciplined, quantifiable approach to the development, operation, and maintenance of software; that is, the application of engineering to software.
The Four Layers of Software Engineering (Pressman)

The four layers rest on a quality focus (bedrock). The process layer provides the framework; methods provide the technical how-to knowledge; tools provide semi-automated support.

1.4 Generic Software Process Framework

A process framework defines a small number of framework activities applicable to all software projects. Pressman's generic framework has five activities:

Five Generic Framework Activities
Umbrella Activities (run across all framework activities)
  • Software project tracking and control — monitor progress against the plan
  • Risk management — assess risks that may affect outcome or quality
  • Software quality assurance (SQA) — activities to ensure quality
  • Technical reviews — assess work products to uncover and remove errors
  • Software configuration management (SCM) — manage change throughout lifecycle
  • Reusability management — define criteria for work product reuse
  • Work product preparation and production — activities to create models, documents, logs, forms, and lists
  • Measurement — define and collect process, project, and product measures

Process Models

Waterfall · Incremental · RAD · Evolutionary · Unified · Concurrent · Component-based · Formal methods
2.1 Prescriptive Process Models
Definition
A software process model is an abstract representation of a software process — a description from a particular perspective of a process. Prescriptive process models advocate an orderly approach to SE.
Waterfall Model (Classic Life Cycle)

Oldest and most widely used paradigm. Demands a systematic, sequential approach. Each phase must be completed before the next begins.

Waterfall Model — Sequential Phases
Problems with Waterfall
Real projects rarely follow the sequential flow; requirements are difficult to state completely at the start; a working version is not available until late in the project; it is inflexible to change.
Incremental Process Model

Combines linear and parallel flow. Multiple sequences are applied with each sequence producing a deliverable increment. Earlier increments serve as basic versions; later increments add capabilities.

Key Idea
The first increment is often a core product. Subsequent increments add features based on feedback. This reduces risk and allows early delivery of partial functionality.
RAD — Rapid Application Development

A high-speed adaptation of the linear sequential model using component-based construction. Suitable for systems that can be modularised. Uses a very short development cycle (60–90 days).

RAD Phases: Business Modeling → Data Modeling → Process Modeling → Application Generation → Testing and Turnover

RAD Disadvantages
Requires sufficient human resources for parallel teams; not appropriate for technically risky or non-modularisable systems; unsuitable when interoperability with existing systems is required.
2.2 Evolutionary Process Models

Evolutionary models are iterative — the engineer develops increasingly complete versions of software. They accommodate changing requirements and deliver working versions early.

Prototyping Model

A prototype is a working model of part of the software. It is used to help stakeholders clarify requirements. The prototype is typically discarded after requirements are clear (throwaway prototyping).

Prototyping Paradigm
Spiral Model (Boehm, 1988)

The spiral model is a risk-driven process model. It couples the iterative nature of prototyping with the controlled, systematic aspects of the waterfall model. Each loop of the spiral represents a phase of the process.

Spiral Model — Four Quadrants
  • Quadrant 1 (Planning): Determine objectives, alternatives, constraints
  • Quadrant 2 (Risk Analysis): Evaluate alternatives, identify & resolve risks
  • Quadrant 3 (Engineering): Develop next-level product
  • Quadrant 4 (Customer Evaluation): Assess results, plan next iteration
Concurrent Development Model

All activities exist simultaneously but in different states. Uses a network of activities that are triggered by events. Activities such as analysis, design, and coding are all potentially active at the same time.

2.3 Unified Process (UP)

The Unified Process (UP), popularised by Ivar Jacobson, Grady Booch, and James Rumbaugh, is a use-case driven, architecture-centric, iterative and incremental software process framework.

PhasePurposePrimary Workflows
InceptionEstablish the business case; identify scopeBusiness modeling, requirements
ElaborationAnalyse the problem domain; establish architectural foundationRequirements, analysis & design
ConstructionDevelop the system incrementallyImplementation, testing
TransitionTransfer the system to usersDeployment, configuration
Key UP Concept
UP organises work into disciplines (Business Modeling, Requirements, Analysis & Design, Implementation, Testing, Deployment) which are active to varying degrees in each phase, visualised as the "hump diagram."
2.4 Component-Based Development (CBD)

CBD incorporates pre-packaged software components (COTS — Commercial Off-The-Shelf) into software. The process emphasises finding existing components and integrating them rather than building from scratch.

  • Identify candidate components from a component repository
  • Evaluate and select components that meet requirements
  • Design the architecture to accommodate selected components
  • Integrate components and conduct regression testing
  • Deploy and deliver the composite system
    • 2.5 Formal Methods Model

    The formal methods model encompasses activities that lead to a mathematical specification of computer software. Formal methods allow a software engineer to specify, develop, and verify a computer-based system using rigorous mathematical notation.

    Advantages
    • Rigorous, unambiguous specification
    • Can prove correctness mathematically
    • Detects inconsistencies/ambiguities early
    Disadvantages
    • Requires specialised mathematical training
    • Not cost-effective for all systems
    • Difficult to model non-functional requirements

    Agile Development

    Agile Manifesto · Scrum · XP · Kanban · Agile principles · Velocity · Sprint planning
    3.1 What is Agility?
    Agility Defined
    Agility means effective response to change. An agile team is able to respond to change quickly — through continuous feedback, close customer collaboration, self-organisation, and lightweight but sufficient processes.

    Agility principles (Pressman's 12 criteria):

    1. Customer satisfaction through early and continuous delivery of valuable software
    2. Welcome changing requirements, even late in development
    3. Deliver working software frequently (weeks rather than months)
    4. Business people and developers work together daily
    5. Build projects around motivated individuals; give them the environment and support they need
    6. Face-to-face conversation is the most efficient form of communication
    7. Working software is the primary measure of progress
    8. Sustainable development pace — sponsors, developers, and users should be able to maintain a constant pace indefinitely
    9. Continuous attention to technical excellence and good design enhances agility
    10. Simplicity — the art of maximising the amount of work not done — is essential
    11. The best architectures, requirements, and designs emerge from self-organising teams
    12. At regular intervals, the team reflects on how to become more effective
    3.2 Agile Manifesto (2001)
    Four Core Values
    • Individuals and interactions over processes and tools
    • Working software over comprehensive documentation
    • Customer collaboration over contract negotiation
    • Responding to change over following a plan

    Note: The items on the right are valued, but those on the left are valued more.

    3.3 Extreme Programming (XP)

    XP is the most widely used agile process. It organises software development around the activities: planning → designing → coding → testing.

    XP Planning (The Planning Game)
    • Customer creates user stories — short descriptions of required functionality
    • XP team assesses each story and assigns a cost (in development weeks)
    • Stories are grouped into releases (small increments)
    • Each release contains iterations (1–3 weeks each)
    XP Core Practices
    PracticeDescription
    Pair ProgrammingTwo programmers work at one workstation. One writes code ("driver"), the other reviews ("navigator"). Roles rotate.
    Test-Driven Development (TDD)Write unit tests before writing code. Tests guide implementation and act as specification.
    RefactoringContinuously restructure existing code to improve design without changing observable behaviour.
    Continuous IntegrationCode is integrated and tested multiple times per day. Integration errors are detected quickly.
    Simple DesignThe simplest design that passes all tests. No speculative generality — YAGNI (You Aren't Gonna Need It).
    On-Site CustomerA real customer representative is part of the team and available full time for questions.
    Collective Code OwnershipAny developer can modify any part of the codebase at any time — no code ownership.
    Coding StandardsAll code follows agreed conventions to support collective ownership.
    40-Hour WeekOvertime two consecutive weeks signals a problem. Sustainable pace protects quality.
    Small ReleasesReleases should be as small as possible — every feature available to customers sooner.
    System MetaphorA simple shared story that describes how the whole system works.
    Planning GameRapid determination of scope of the next release by combining business priorities and technical estimates.
    3.4 Scrum

    Scrum is a framework within which people can address complex adaptive problems, while productively and creatively delivering products of the highest possible value.

    Scrum Roles
    RoleResponsibilities
    Product OwnerRepresents stakeholders; owns and prioritises the Product Backlog; defines acceptance criteria; maximises product value
    Scrum MasterServant-leader for the team; removes impediments; facilitates Scrum events; protects the team from external interference; coaches on Scrum
    Development TeamCross-functional, self-organising team (3–9 people); responsible for delivering a potentially shippable product increment each sprint
    Scrum Artifacts
    • Product Backlog: Ordered list of everything that might be needed in the product. The Product Owner is responsible for its content and ordering.
    • Sprint Backlog: Set of Product Backlog items selected for the Sprint plus a plan for delivering the product Increment and achieving the Sprint Goal.
    • Increment: Sum of all Product Backlog items completed during a Sprint plus the value of all previous increments. Must be "Done" and usable.
    Scrum Events (Ceremonies)
    Scrum Sprint Flow
    EventDurationPurpose
    Sprint1–4 weeksTime-box in which a "Done" increment is created
    Sprint Planning≤8 hrs/4-week sprintTeam plans the work for the Sprint; Sprint Goal defined
    Daily Scrum (Stand-up)15 minutesSynchronise activities; inspect progress; identify impediments
    Sprint Review≤4 hrs/4-week sprintDemonstrate increment to stakeholders; inspect and adapt backlog
    Sprint Retrospective≤3 hrs/4-week sprintInspect team process; create improvement plan for next sprint
    Scrum Velocity and Burndown

    Velocity is the amount of work a team can complete in a single sprint, measured in story points. Used for sprint planning and release forecasting.

    Burndown chart tracks remaining work (story points) against time in a sprint. Ideal line shows linear progress; actual line reveals impediments.

    3.5 Kanban

    Kanban is a lean, flow-based approach to software development. It is not prescriptive about roles, events, or time-boxes.

    Core Kanban Principles
    • Visualise work: All work items are visible on a Kanban board
    • Limit WIP (Work In Progress): Each column has a maximum number of items — prevents bottlenecks
    • Manage flow: Monitor the flow of work; measure cycle time and lead time
    • Make process policies explicit
    • Implement feedback loops: Regular cadences for retrospective and review
    • Improve collaboratively, evolve experimentally

    Professional Ethics in Software Engineering

    SE Code of Ethics · ACM/IEEE Code · Professional responsibility · Privacy · IP · Safety
    28.1 What is Professional Ethics?
    Definition
    Professional ethics concerns the moral obligations and responsibilities of professionals towards their clients, employers, colleagues, the profession, and society at large. For software engineers, it encompasses integrity, competence, honesty, and the public good.

    Because software permeates every aspect of modern life — healthcare, finance, transportation, communication — the ethical choices of software engineers have profound societal implications.

    28.2 ACM/IEEE Software Engineering Code of Ethics (8 Principles)

    The ACM/IEEE-CS Joint Task Force developed the Software Engineering Code of Ethics and Professional Practice. It defines eight principles:

    #PrincipleCore Obligation
    1PUBLICAct consistently with the public interest. Software engineers shall place the safety, health and welfare of the public above all other considerations.
    2CLIENT AND EMPLOYERAct in a manner that is in the best interests of client and employer, consistent with the public interest.
    3PRODUCTEnsure that their products and related modifications meet the highest professional standards possible. Software engineers shall produce quality work.
    4JUDGMENTMaintain integrity and independence in their professional judgment. Avoid deception; not misrepresent work.
    5MANAGEMENTSoftware engineering managers and leaders shall subscribe to and promote an ethical approach. Managers must not require engineers to violate the code.
    6PROFESSIONAdvance the integrity and reputation of the profession, consistent with the public interest.
    7COLLEAGUESBe fair to and supportive of colleagues. Assist fellow engineers in adhering to the code.
    8SELFParticipate in lifelong learning. Continuing education; maintain current technical competence; improve own ability to produce quality work.
    Mnemonic
    PC PM PC S — Public, Client, Product, Management, Profession, Colleagues, Self (or use PCPMJPC-S)
    28.3 Key Ethical Issues in Software Engineering
    Intellectual Property (IP)

    Software is protected by copyright (automatically), patents (explicit registration), and trade secrets. Ethical obligations include:

    • Not copying or distributing software without proper license
    • Respecting open-source licenses (GPL, MIT, Apache)
    • Not reverse-engineering proprietary software without authorisation
    • Properly attributing third-party code and libraries
    Privacy and Data Protection

    Software engineers who build systems that collect, process, or store personal data have obligations to:

    • Collect only necessary data (data minimisation)
    • Protect data against unauthorised access (confidentiality)
    • Allow users to know, correct, and delete their data (data subject rights)
    • Disclose data breaches promptly
    • Comply with regulations: GDPR (Europe), IT Act (India), HIPAA (USA healthcare)
    Computer Crime and Misuse

    Ethical software engineers do not:

    • Develop malware, spyware, or ransomware
    • Access computer systems without authorisation
    • Build systems intended to defraud or harm users
    • Assist in surveillance without legal authority
    Safety-Critical Systems

    Software engineers working on systems where failures could cause death or serious injury (medical devices, aircraft control, nuclear systems) have heightened ethical obligations:

    • Rigorous verification and validation
    • Conservative design — fail-safe defaults
    • Transparency about known limitations and failure modes
    • Not certifying software as safe without proper evidence
    Notable Case: Therac-25
    A radiation therapy machine (1985–87) that delivered massive radiation overdoses due to software race conditions, killing several patients. Illustrates the life-or-death consequences of software quality failures and the ethical obligations of software engineers.
    Whistleblowing

    If an engineer believes an employer is violating the code of ethics and internal remedies have failed, they may be obligated to report to external authorities (whistleblowing). The code places public welfare above employer loyalty when public safety is at risk.

    AI and Algorithmic Ethics
    • Bias and fairness: ML systems may encode historical biases that discriminate against protected groups
    • Transparency: Users and affected parties should understand how automated decisions are made
    • Accountability: Engineers are responsible for the systems they build even after deployment
    • Consent: Data used to train AI should be obtained with proper consent
    28.4 Professional Competence and Responsibility

    A software engineer must:

    • Only accept work they are competent to perform
    • Acknowledge when they need help or are outside their expertise
    • Maintain professional development through continuous learning
    • Be honest about estimates, deadlines, and technical limitations
    • Not agree to unrealistic schedules or quality compromises under management pressure
    Golden Rule
    Do not produce software that you would be unwilling to take responsibility for. If you would be embarrassed or ashamed to have your name associated with the software, it probably shouldn't be shipped.

    Understanding Requirements

    RE tasks · Elicitation techniques · Problem analysis · Validation · Negotiation · Requirement types
    4.1 Requirements Engineering
    Definition
    Requirements Engineering (RE) is a systematic process of developing requirements through an iterative cooperative process of analysing the problem, documenting the resulting observations in a variety of representation formats, and checking the accuracy of the understanding gained.

    RE sits at the interface between software engineering and systems engineering. Poor requirements are the most common cause of project failure.

    Requirements Engineering Bridge
    4.2 Types of Requirements
    TypeDefinitionExamples
    Functional Describe what the system shall do — specific behaviour, functions, or services "The system shall allow users to place an order"; "The system shall generate monthly invoices"
    Non-Functional (NFR) Constraints on the system or its development — quality attributes Performance, security, availability, usability, maintainability
    Domain Requirements from the application domain that reflect characteristics and constraints of the domain Medical: "The system shall comply with HL7 standards"; Banking: "Audit logs must be retained for 7 years"
    User Requirements Natural language statements of what the system should provide and the constraints under which it operates "Users should be able to search for products by name or category"
    System Requirements Detailed, structured description of the functions, services, and constraints — forms the basis for design Formal SRS with precise, testable, unambiguous statements
    4.3 Requirements Engineering Tasks
  • Inception: Understand the problem, who wants the solution, and why. Establish the basic understanding of the problem and stakeholders.
  • Elicitation: Gather requirements from all stakeholders using various techniques. The most critical and difficult step.
  • Elaboration: Expand and refine the initial requirements. Develop analysis models (use cases, activity diagrams).
  • Negotiation: Resolve conflicts between stakeholder requirements. Prioritise and agree on scope.
  • Specification: Document requirements in a formal Software Requirements Specification (SRS) using IEEE 830.
  • Validation: Review specifications to ensure correctness, consistency, completeness, and verifiability.
  • Management: Track and control changes to requirements throughout the project lifecycle.
    • 4.4 Requirements Elicitation Techniques
    TechniqueDescriptionBest Used When
    InterviewsStructured or unstructured conversations with stakeholders. Structured: predefined questions. Unstructured: open conversation.Gathering detailed information from individual stakeholders
    Questionnaires / SurveysWritten questions distributed to many stakeholders simultaneouslyLarge number of geographically dispersed stakeholders
    Observation (Ethnographic)Analyst observes users in their actual work environmentUnderstanding implicit/tacit requirements that users cannot articulate
    Workshops (JAD)Joint Application Development — facilitated group sessions with stakeholders and developersResolving conflicts; rapid requirements gathering with diverse groups
    BrainstormingUnconstrained group idea generation — no criticism during idea generation phaseCreative exploration; generating innovative feature ideas
    PrototypingBuild a working mock-up; stakeholders interact with it and provide feedbackWhen requirements are unclear or users have difficulty articulating needs
    Document AnalysisExamine existing documents, manuals, procedures, reports, and legacy system documentationReplacement systems; regulatory compliance requirements
    Use Cases / User StoriesScenario-based descriptions of system interactions from the user's perspectiveAgile projects; understanding user goals and workflows
    StoryboardingVisual sequence of sketches showing user interactions with the systemUI-heavy applications; communicating with non-technical stakeholders
    Focus GroupsModerated group discussion with representative usersMarket research; product feature prioritisation
    Elicitation Problems
    • Stakeholders don't know what they want — Tacit knowledge; they know it when they see it
    • Stakeholders express requirements in their own terms — Domain knowledge gap between users and engineers
    • Different stakeholders may have conflicting requirements — Negotiation required
    • Organisational and political factors — Requirements may reflect politics, not real needs
    • Requirements change — Business environment changes; stakeholder understanding evolves
    4.5 Requirements Validation

    Validation checks that the requirements define the system the customer actually wants. Distinct from verification (which checks that the system conforms to its specification).

    Validation vs Verification
    Validation: "Are we building the right product?" (Does the specification match stakeholder needs?)
    Verification: "Are we building the product right?" (Does the system conform to its specification?)
    Validation CheckWhat It Tests
    ValidityDoes the system provide functions that best support stakeholder needs?
    ConsistencyAre there any requirements conflicts?
    CompletenessAre all functions required by the customer included?
    RealismCan the requirements be implemented given the available budget and technology?
    VerifiabilityCan each requirement be checked?
    ComprehensibilityIs the specification properly understood?
    TraceabilityIs the origin of each requirement clearly stated?
    AdaptabilityCan a requirement be changed without large-scale impact?
    4.6 Requirements Negotiation

    When multiple stakeholders have conflicting requirements, the SE team must negotiate. Effective negotiation:

    • Identifies stakeholders and their relative authority/priority
    • Discovers hidden assumptions behind stated requirements
    • Uses formal prioritisation techniques (MoSCoW: Must-have, Should-have, Could-have, Won't-have)
    • Seeks compromise — often through phased delivery or scope reduction
    • Documents agreed requirements with stakeholder sign-off

    5.1 Analysis Models Overview
    Definition
    Requirements models (analysis models) provide a representation of information, function, and behaviour of software. They bridge the gap between imprecise informal requirements and precise design. They are created during elaboration in the RE process.

    Requirements modeling uses four model types:

    Model TypeNotationPurpose
    Scenario-basedUse cases, user storiesHow users interact with the system
    Class-basedClass diagrams, CRC cardsObjects, attributes, relationships
    BehaviouralState diagrams, sequence diagramsHow the system responds to events
    Flow-orientedDFDs, activity diagramsHow data transforms through the system
    5.2 Scenario-Based Modeling — Use Cases

    A use case describes a sequence of actions that provides a measurable value to an actor. It represents the system's functional behaviour from the user's perspective.

    Use Case Components
    ElementDescription
    ActorA role played by a person or system that interacts with the subject. Drawn as a stick figure.
    Use CaseA description of a set of sequences of actions performed by the system. Drawn as an ellipse.
    AssociationA line between actor and use case showing participation
    <<include>>One use case always includes the behaviour of another
    <<extend>>One use case optionally extends the behaviour of another under certain conditions
    GeneralisationA use case inherits behaviour from a parent use case
    Use Case Template (Cockburn Format)
    Use Case Template Elements
    Use Case Name | ID | Actor(s) | Preconditions | Main Success Scenario (Basic Flow) | Extensions (Alternative Flows) | Postconditions | Priority

    5.3 Class-Based Modeling
    UML Class Diagram

    A class diagram shows the static structure of a system — the classes (objects), their attributes, operations, and the relationships between them.

    RelationshipSymbolMeaning

    CRC Cards (Class-Responsibility-Collaborator)

    Class Name: Order

    5.4 Behavioural Modeling
    State Diagram (State Machine)

    Shows the dynamic behaviour of an object through its states and transitions. Useful for event-driven systems, UI behaviour, and protocol design.

    • State: A condition or situation during the life of an object (rectangle with rounded corners)
    • Transition: A change from one state to another, triggered by an event (arrow with label)
    • Initial state: Filled black circle
    • Final state: Filled circle inside a larger circle
    • Guard: A boolean condition that must be true for a transition to occur [in square brackets]
    • Action: Activity that occurs during a transition or in a state
    Sequence Diagram

    Shows how objects interact in time — the sequence of messages exchanged between objects to carry out a use case.

    ElementNotationMeaning
    Object/ActorRectangle at top with nameParticipant in the interaction
    LifelineDashed vertical line below objectObject's existence over time
    Activation barNarrow rectangle on lifelinePeriod when object is executing
    Synchronous messageSolid arrow → (filled head)Caller waits for return
    Asynchronous messageSolid arrow → (open head)Caller does not wait
    Return messageDashed arrow ←Return value/control from called operation
    Activity Diagram

    Shows the flow of control within a process — similar to a flowchart but supports parallel activities (fork/join bars). Useful for modelling workflows, algorithms, and use case steps.

    • Action: Rounded rectangle — a single step in the activity
    • Decision node: Diamond — branching based on a condition
    • Fork/Join: Thick horizontal bar — parallel execution (fork splits; join synchronises)
    • Swimlanes: Partition the diagram by actor/responsibility
    5.5 Data Modeling (Flow-Oriented)
    Data Flow Diagram (DFD)

    A DFD shows the flow of data through a system, the processes that transform data, and the data stores. It is a functional decomposition tool.

    SymbolElementDescription
    Circle / bubbleProcessTransforms input data into output data
    ArrowData FlowData moving between elements; labelled with data name
    RectangleExternal EntitySource or sink of data outside the system
    Open-ended rectangleData StoreRepository where data is stored (database, file)
    DFD Levelling
    Level 0 (Context Diagram): Single process representing entire system with all external entities
    Level 1: Major processes and data flows; decomposition of Level 0
    Level n: Each process further decomposed until primitive (non-decomposable) processes are reached
    Data Dictionary

    A data dictionary is a structured repository that defines all data elements used in DFDs — their names, aliases, composition, values, and meanings. It ensures consistent terminology across the team and documentation.

    Entity-Relationship Diagram (ERD)

    An ERD models the data perspective of a system — the entities (things/objects), attributes (properties), and relationships between entities.

    • Entity: A person, place, thing, or event about which data is stored (rectangle)
    • Attribute: Property of an entity (ellipse)
    • Relationship: Association between entities (diamond)
    • Cardinality: 1:1, 1:N, M:N — how many entity instances participate

    6.1 Software Requirements Specification (SRS)
    Definition (IEEE 830)
    An SRS is a description of the software system to be developed. It lays out functional and non-functional requirements, and may include a set of use cases that describe user interactions that the software must provide.
    IEEE 830 SRS Structure
    1. Introduction: Purpose, scope, definitions/acronyms, references, overview
    2. Overall Description: Product perspective, product functions, user characteristics, constraints, assumptions & dependencies
    3. Specific Requirements:
      • Functional requirements (each function described in detail)
      • External interface requirements (UI, hardware, software, communications)
      • Non-functional requirements (performance, security, reliability, etc.)
      • Design constraints
      • Other requirements
    4. Appendices: Glossary, analysis models, issues list
    Characteristics of a Good SRS (IEEE 830)
    CharacteristicDescription
    CorrectEvery stated requirement is one that the software shall possess
    UnambiguousEvery requirement has one and only one interpretation
    CompleteAll significant requirements are included; all possible responses defined
    ConsistentNo subset of requirements conflict with each other
    Ranked for importanceEach requirement is identified as essential, conditional, or optional
    VerifiableThere exists a finite cost-effective process by which it can be checked
    ModifiableStructure allows changes easily, completely, and consistently
    TraceableOrigin of each requirement is clear; each requirement can be referenced in future documentation
    6.2 Non-Functional Requirements (NFR) in Depth

    NFRs define system qualities and constraints — they constrain HOW the system provides its functional requirements.

    NFR CategoryDescriptionExample Metric
    PerformanceResponse time, throughput, capacity"95th percentile response <2s under 1000 users"
    ReliabilityAvailability, MTBF, MTTR, fault tolerance"99.9% uptime; MTTR <30 min"
    SecurityAuthentication, authorisation, confidentiality, integrity"AES-256 encryption; OAuth 2.0 auth"
    UsabilityLearnability, efficiency, memorability, error recovery"Task completion in <3 min for new user"
    MaintainabilityEase of change, modularity, testability"McCabe complexity <10 per function"
    PortabilityAdaptability, installability, replaceability"Runs on Windows, macOS, Linux without recompile"

    6.4 Requirements Change Management

    Requirements change for many reasons: stakeholder understanding evolves, business priorities shift, technology changes, regulatory updates. A change management process prevents uncontrolled change (scope creep).

  • Change request submitted — Any stakeholder can submit a change request using a formal form
  • Impact analysis — SE team analyses impact on cost, schedule, other requirements, architecture
  • Change Review Board (CRB) — Stakeholders and SE lead decide: approve, reject, or defer
  • SRS updated — If approved, SRS is formally revised with version control
  • Downstream artifacts updated — Design, code, tests updated via RTM
  • Stakeholders notified — All affected parties informed of the approved change
    • Scope Creep
    Uncontrolled changes or continuous growth in project scope — the most common cause of project overruns. Prevented by formal change management, version control, and stakeholder agreement on baselines.

    9.1 The Design Process
    Software Design
    Software design is the process of defining the architecture, components, interfaces, and other characteristics of a system or component. It translates requirements into a blueprint for constructing software.

    Design occurs at two levels: Architectural design (high-level structure of software) and Component-level design (detailed algorithms and data structures for each module).

    Design as a Translation Process
    9.2 Fundamental Design Concepts
    Abstraction

    Abstraction allows us to focus on essential characteristics while ignoring irrelevant details. Two forms:

    • Procedural abstraction: A sequence of instructions with a specific and limited function (e.g., a function that opens a file)
    • Data abstraction: A collection of data that describes a data object (e.g., a "Document" object with attributes title, content, author)
    Architecture

    Software architecture is the overall structure of the software — the organisation of software components (modules), their properties, and the relationships between them. It is the first design decision and the hardest to change.

    Patterns

    A design pattern is a recurring solution to a standard problem in a given context. Patterns capture expert design knowledge and make it reusable.

    • Architectural patterns: MVC, Layered, Microservices, Event-Driven
    • Design patterns (GoF — Gang of Four): Creational (Singleton, Factory), Structural (Adapter, Decorator, Façade), Behavioural (Observer, Strategy, Command)
    Separation of Concerns

    A software design principle: any complex problem can be more easily handled if decomposed into pieces that can be solved independently. Examples:

    • Separating UI from business logic from data access (MVC/3-tier)
    • Separating error handling from normal flow (aspect-oriented programming)
    • Separating configuration from code
    Modularity

    Software is divided into separately named and addressable components (modules) that are integrated to satisfy requirements. Modularity:

    • Reduces complexity (divide and conquer)
    • Allows parallel development
    • Facilitates testing and maintenance
    • Supports reuse
    Modularity Cost Trade-off
    As number of modules increases, effort per module decreases (divide and conquer advantage) BUT the effort to integrate modules increases. There is an optimal module count that minimises total effort.
    Information Hiding (Encapsulation)

    Each module knows only what it needs to know. Implementation details are hidden behind interfaces. Proposed by David Parnas (1972). Reduces the impact of change — a change inside a module does not propagate to other modules that use it.

    Stepwise Refinement (Wirth)

    Top-down design strategy. Start with a high-level statement and elaborate it step by step into lower-level detail. Complementary to abstraction — abstraction goes upward; refinement goes downward.

    9.3 Functional Independence — Cohesion and Coupling

    Functional independence is the single most important design concept. A module achieves functional independence when it performs a single function and has little interaction with other modules.

    Measured by two criteria: cohesion and coupling.

    Cohesion (Higher is Better)

    Cohesion measures how closely related the functions within a single module are. A highly cohesive module does one thing well.

    TypeDescriptionQuality
    FunctionalModule performs a single, well-defined function (e.g., calculateSquareRoot())Best
    SequentialOutput of one function is input to the nextGood
    CommunicationalFunctions access or operate on the same dataAcceptable
    ProceduralFunctions are related by sequence of control flowModerate
    TemporalFunctions related by timing (initialisation module)Weak
    LogicalFunctions logically related but not functionally (e.g., all I/O in one module)Poor
    CoincidentalFunctions have no meaningful relationshipWorst
    Coupling (Lower is Better)

    Coupling measures the degree of interdependence between modules. Low coupling means a change in one module has minimal impact on others.

    TypeDescriptionQuality
    Data couplingModules share data through parameters only (simple data items)Best
    Stamp couplingModules share a composite data structure; only part of it is usedGood
    Control couplingOne module passes a control flag that directs logic in anotherModerate
    External couplingModules share an externally imposed format or interfaceModerate
    Common couplingModules share global dataPoor
    Content couplingOne module modifies data inside another moduleWorst
    Design Goal: High Cohesion + Low Coupling
    9.4 Design Model Elements
    Design ElementPurposeNotation
    Data designTransforms information domain models into data structures (classes, databases)Class diagrams, ERDs, schema
    Architectural designDefines overall structure — subsystems and their interactionsComponent diagrams, deployment diagrams
    Interface designDescribes how modules communicate with each other, user, and external systemsInterface specifications, wireframes
    Component designDescribes internal processing logic of each componentFlowcharts, pseudocode, PDL (Program Design Language)
    9.5 Refactoring
    Definition
    Refactoring is the process of changing a software system in such a way that it does not alter the external behaviour of the code yet improves its internal structure. It is a disciplined technique for cleaning up code.

    Common refactoring operations: Extract Method, Rename Variable, Remove Duplicate Code, Replace Magic Number with Named Constant, Extract Class, Move Method, Introduce Polymorphism.

    Refactoring reduces technical debt — the cost of rework caused by choosing an easy solution now instead of a better approach that would take longer.

    Architectural Design

    Architectural styles · Layered · MVC · Client-server · Microservices · Pipe-and-filter · Component & connector · Quality attributes
    10.1 What is Software Architecture?
    Definition (Bass, Clements, Kazman)
    The software architecture of a program or computing system is the structure or structures of the system, which comprise software elements, the externally visible properties of those elements, and the relationships among them.

    Architecture decisions are made early and are difficult to change. They have the greatest impact on quality attributes (performance, security, maintainability) of the final system.

    Why Architecture Matters
    • Enables communication among stakeholders about system structure
    • Highlights early design decisions with widest impact
    • Constitutes a transferable, reusable model (architectural patterns)
    • Constrains implementation — sets boundaries within which modules must work
    • Enables analysis of quality attributes before a line of code is written
    10.2 Architectural Styles (Patterns)
    Layered Architecture

    The system is organised into layers, each providing services to the layer above and consuming services from the layer below. Each layer has a well-defined responsibility.

    Layered Architecture (4-Tier Web Application)
    Advantages
    • Clear separation of concerns
    • Easy to understand and develop incrementally
    • Individual layers can be replaced
    Disadvantages
    • Performance overhead (crossing layers)
    • Difficult to keep layers clean when requirements evolve
    • Not ideal for high-performance systems
    MVC — Model-View-Controller
    MVC Architecture Pattern
    • Model: Encapsulates data and business rules. Notifies View when data changes.
    • View: Renders the model data for the user. Multiple views can display the same model.
    • Controller: Handles user input; updates the model; selects the view to display.
    Client-Server Architecture

    System is partitioned into clients (requestors of services) and servers (providers of services). Servers provide services to multiple clients. Communication typically over a network using request/response protocols.

    • 2-tier: Client + Database server
    • 3-tier: Presentation + Application server + Database server
    • N-tier: Multiple tiers for scalability and separation
    Pipe-and-Filter Architecture

    Components (filters) process data streams; data flows between components through pipes. Each filter is independent and transforms its input. Suitable for data transformation pipelines.

    Pipe-and-Filter Pattern
    Repository (Data-Centred) Architecture

    A central data store (repository/blackboard) is shared by all subsystems. Components access and modify data in the repository. Suitable for systems with complex data and many components that need shared access (IDEs, CAD systems, AI expert systems).

    Microservices Architecture

    Application is composed of small, independently deployable services, each running in its own process and communicating through APIs (REST, gRPC, message brokers). Each service is responsible for a specific business capability.

    PrincipleDescription
    Single ResponsibilityEach service owns one business capability
    Loose couplingServices communicate via well-defined APIs; no shared database
    Independent deploymentEach service can be deployed, scaled, and updated independently
    Technology diversityEach service can use its own language, framework, and database
    Decentralised dataEach service manages its own data store
    Event-Driven Architecture

    Components communicate by producing and consuming events through a message broker (Kafka, RabbitMQ). Producers publish events without knowing consumers. Enables loose coupling, scalability, and real-time processing.

    10.3 Architecture and Quality Attributes

    Different architectural styles support different quality attributes. The architect must select styles that meet the system's most critical quality requirements.

    Quality AttributeArchitectural Tactics / Styles
    PerformanceCaching, load balancing, async messaging, CDN, connection pooling
    AvailabilityRedundancy, failover, circuit breakers, health monitoring, multi-AZ deployment
    SecurityAPI gateway (centralised auth), layered security, zero-trust, service mesh
    ModifiabilityLayered, microservices, high cohesion/low coupling
    ScalabilityStateless services, horizontal scaling, event-driven, microservices
    TestabilitySeparation of concerns, dependency injection, interfaces over implementations

    11.1 What is a Component?
    Definition
    A component is a modular, deployable, and replaceable part of a system that encapsulates implementation and exposes a set of interfaces. It is a building block of software that provides a cohesive set of behaviours.

    In OO terms, a component can be a class, a set of classes, or a subsystem. In component-based systems, it is a unit that can be independently deployed and composed.

    ViewWhat a Component Is
    OO viewA class or cluster of collaborating classes with well-defined interfaces
    Conventional viewA functional element (module) of a program that processes data
    Process viewA pipeline element that transforms data as it flows
    11.2 Design Principles for Components (SOLID)
    PrincipleStatementConsequence
    S — Single Responsibility (SRP) A class should have only one reason to change High cohesion; each class has one job
    O — Open/Closed (OCP) Software entities should be open for extension but closed for modification New features added via inheritance/composition, not by modifying existing code
    L — Liskov Substitution (LSP) Objects of a subclass should be substitutable for objects of the superclass Inheritance hierarchies must be semantically correct — no surprising overrides
    I — Interface Segregation (ISP) Clients should not be forced to depend on methods they do not use Prefer many small, specific interfaces over one large general-purpose interface
    D — Dependency Inversion (DIP) High-level modules should not depend on low-level modules. Both should depend on abstractions. Depend on interfaces/abstractions, not concrete implementations — enables testing and flexibility
    Additional Package Principles (Robert Martin)
    REP (Reuse-Release Equivalence): Unit of reuse = unit of release
    CCP (Common Closure): Classes that change together belong together
    CRP (Common Reuse): Classes that are used together belong together
    ADP (Acyclic Dependencies): No cycles in the package dependency graph
    11.3 Conducting Component-Level Design

    Component-level design translates the architectural and requirements models into detailed design specifications for each component.

  • Identify design classes corresponding to infrastructure domain
  • Specify message detail for all classes that correspond to analysis classes
  • Identify appropriate interfaces for each design class
  • Elaborate attributes and define data types and structures
  • Describe processing flow for each operation — use pseudocode/PDL or activity diagrams
  • Describe persistent data sources (databases, files) and identify required classes to manage them
  • Develop and simplify component-level design representation; apply design principles and refactor
  • Review the design against requirements and architectural model
    • 11.4 Program Design Language (PDL) / Pseudocode

    PDL (Program Design Language) is a formal pseudocode used to describe component algorithms without a specific programming language syntax. It uses structured programming constructs:

    PDL Construct Examples

    PDL uses: IF/THEN/ELSE/ENDIF, WHILE/DO/ENDWHILE, FOR EACH/DO/ENDFOR, PROCEDURE, RETURN, EXCEPTION HANDLING

    12.1 The Importance of UI Design
    Definition
    User Interface (UI) design focuses on the communication between human and computer. User Experience (UX) is broader — it encompasses all aspects of end-user interaction with a company, its services, and its products.

    A well-designed UI is the single most important factor in user satisfaction and adoption. Even technically excellent software fails if users cannot operate it effectively.

    UI Design Principle (Pressman)
    The best interface is one where users never have to think about the interface — it is transparent and allows them to focus entirely on their task.
    12.2 Three Golden Rules of UI Design (Shneiderman)
    Shneiderman's Three Golden Rules
    • Place the user in control — The user should feel that they are in charge of the interface, not the other way around
    • Reduce the user's memory load — The interface should not require users to remember information from one screen to another
    • Make the interface consistent — Similar interactions should behave similarly across the entire system
    Expanding the Golden Rules — Shneiderman's 8 Golden Rules
    #RuleDescription
    1Strive for consistencyConsistent sequences of actions, identical terminology, consistent commands across similar situations
    2Enable frequent users to use shortcutsAbbreviations, function keys, hidden commands for expert users to speed up interaction
    3Offer informative feedbackSystem feedback for every user action — minor feedback for frequent/minor actions; major feedback for infrequent/major actions
    4Design dialogs to yield closureSequences of actions organised into groups with a beginning, middle, and end. Completion notification gives satisfaction.
    5Offer error prevention and simple error handlingDesign so users cannot make serious errors. If error occurs: detect it and offer simple, understandable recovery
    6Permit easy reversal of actions (Undo)Actions should be reversible where possible. Relieves anxiety and encourages exploration.
    7Support internal locus of controlExperienced users want to feel in control. Avoid surprises; respond to user actions, not vice versa
    8Reduce short-term memory loadHuman short-term memory capacity is 7±2 items. Interfaces should not demand more.
    12.3 UI Design Process
  • User, task, and environment analysis: Who are the users? What tasks do they perform? What is their technical expertise? What environment will they use the system in?
  • Interface design: Define the interaction model (menu, form, direct manipulation); design metaphors; define interface objects and actions; design the screen layout and navigation structure
  • Prototyping: Build low-fidelity (paper, wireframe) then high-fidelity prototypes to evaluate design choices before implementation
  • Evaluation: Heuristic evaluation, usability testing, A/B testing. Assess the interface design against usability criteria.
  • Refinement and iteration: Apply findings from evaluation to improve the design; repeat prototyping and evaluation
    • User Analysis
    • Novice users: Occasional, non-technical users — need guidance, affordances, visible options, error recovery
    • Knowledgeable intermittent users: Reasonable technical knowledge but don't use the system every day — need reminders, clear terminology
    • Expert frequent users: Experienced power users — need shortcuts, customisation, minimal friction
    • Accessibility: Users with disabilities — visual, motor, cognitive, hearing impairments — must be considered (WCAG standards)

    12.5 Interface Design Elements
    Fitts's Law

    The time to acquire a target is a function of the distance to the target and the size of the target. Larger, closer targets are faster to hit.

    Fitts's Law Formula
    T = a + b log₂(D/W + 1) where T = movement time, D = distance to target, W = width (size) of target. Practical implication: make buttons large and place frequently used actions close to the cursor's natural position.
    Affordance and Signifiers
    • Affordance: The relationship between an object and an agent — what action the object allows (a button affords clicking)
    • Signifier: A perceptible signal that communicates appropriate behaviour (a raised button signals it can be pressed)
    • Mapping: Relationship between controls and their effects (steering wheel maps naturally to direction of travel)
    • Feedback: Information about what has been done and what has happened (button changes colour when clicked)
    • Conceptual model: User's mental model of how the system works
    Visual Design Principles
    PrincipleApplication in UI
    ProximityRelated elements should be grouped close together
    AlignmentElements should be aligned to create visual order and structure
    RepetitionRepeat visual elements (colour, font, shape) to create consistency
    ContrastMake important elements stand out; ensure sufficient colour contrast for accessibility
    HierarchySize, weight, and colour should reflect the importance of elements
    White spaceNegative space reduces clutter and improves focus
    Navigation Design
    • The 3-click rule (guideline, not law): Users should reach any information within 3 clicks
    • Breadcrumbs: Show users their current location in the hierarchy
    • Consistent navigation: Navigation elements in the same location on every page
    • Search: Always provide search for content-rich systems
    • Error pages: 404 pages should help users recover, not just state an error

    12.7 Accessibility (WCAG)

    Web Content Accessibility Guidelines (WCAG 2.1) defines four principles:

    PrincipleMeaningExamples
    PerceivableInformation and UI components must be presentable to users in ways they can perceiveAlt text for images; captions for video; sufficient colour contrast (4.5:1 ratio)
    OperableUI components and navigation must be operableFull keyboard navigation; no keyboard traps; skip links; no content that causes seizures

    12.8 Responsive and Mobile UI Design
    • Mobile-first design: Design for the smallest screen first, then scale up
    • Touch targets: Minimum 44×44 points (Apple HIG) or 48×48 dp (Android) for touch targets
    • Thumb zone: Design interactive elements within comfortable thumb reach on smartphones
    • Fluid grids and flexible images: Use relative units (%, em, rem, vw) instead of fixed pixels
    • Breakpoints: CSS media queries define layout changes at specific screen widths (320px, 768px, 1024px, 1440px typical)
    • Progressive disclosure: Show only necessary information; reveal complexity on demand
    • Offline capability: Progressive Web Apps (PWAs) use service workers to function offline
